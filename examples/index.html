Introduction

Deploying real-time human avatar capture and rendering applications usually requires a hefty Python-based machine learning stack. Libraries such as PyTorch, JAX, and CUDA are standard in research codebases, but they introduce major friction for deployment, particularly in client-side, real-time applications.

The h3d.cpp project aims to consolidate performance capture on workstation-grade machines using C++ and LibTorch, skipping the heavy dependencies of Python. However, to bring these avatars to the web for interactive user experiences, we must solve a different problem: how to efficiently evaluate 3D human body models like SMPL (Skinned Multi-Person Linear model) directly inside the browser.

This article breaks down two distinct ways to solve this problem as implemented in h3d.cpp's web frontend (smpl-web): a high-performance C++ WebAssembly (WASM) CPU engine, and a hybrid WebGL GPU pipeline utilizing Three.js.

The Mathematical Foundation of SMPL

To understand the rendering pipelines, we must first look at the mathematical formulations that define the shape and posture of the SMPL model. SMPL represents a human body as a template mesh \(T \in \mathbb{R}^{N \times 3}\) containing \(N = 6890\) vertices, controlled by shape parameters \(\beta\) (10 principal components) and pose parameters \(\theta\) (24 joints represented in axis-angle format, giving a 72-dimensional vector).

The transformation pipeline proceeds through four main mathematical steps:

1. Shape Blend Shapes

The shape parameters \(\beta\) scale predefined directions of body variation (e.g., height, weight, proportions) to deform the template mesh:

\[T_s(\beta) = T_{template} + B_S(\beta)\]

where \(B_S(\beta) = \sum_{k=1}^{10} \beta_k S_k\) is the shape deformation vector, and \(S_k \in \mathbb{R}^{6890 \times 3}\) represent the shape principal component displacements.

2. Pose Blend Shapes

To avoid artifacts like the "collapsing elbow" during joint rotations, SMPL adds pose-dependent vertex offsets before skinning:

\[T_p(\theta, \beta) = T_s(\beta) + B_P(\theta)\]

where \(B_P(\theta) = \sum_{n=1}^{207} (R_n(\theta) - R_n(\theta^*)) P_n\) is the pose-corrective offset vector, \(R_n(\theta)\) represents the local rotation matrix of joint \(n\), and \(P_n\) represents the pose-corrective displacements. The summation is over \(23 \text{ joints} \times 9 \text{ matrix elements} = 207\) dimensions.

3. Joint Regression

The 3D locations of the 24 joints \(J(\beta)\) are regressed from the shaped mesh template:

\[J(\beta) = \mathcal{J} T_s(\beta)\]

where \(\mathcal{J} \in \mathbb{R}^{24 \times 6890}\) is a sparse regressor matrix learned from data.

4. Linear Blend Skinning (LBS)

Finally, the posed vertices are mapped to their world coordinates using joint transformations recursively computed along the kinematic tree:

\[v_i' = \sum_{j=1}^{24} w_{i, j} G_j(\theta, J(\beta)) v_{i, \text{posed}}\]

where \(w_{i, j}\) is the skinning weight defining the influence of joint \(j\) on vertex \(i\), and \(G_j(\theta, J(\beta)) \in \mathbb{R}^{4 \times 4}\) is the global transformation matrix of joint \(j\) computed by Forward Kinematics (FK).

Linear Blend Skinning Constraints

Each vertex in the SMPL model is typically influenced by multiple bones. To perform LBS efficiently, we restrict the influence to the top 4 most dominant joints per vertex and normalize their weights to sum to 1. This matches the native constraints of GPU hardware skinning engines.

Two Rendering Philosophies on the Web

To implement these equations in a web browser, h3d.cpp provides two architectures inside the interactive viewer:

Pipeline A: WASM CPU Engine

This engine compiles the C++ SMPL implementation directly to WebAssembly using Emscripten. All steps—including Rodrigues rotation matrix computation, Shape Blend Shapes, Pose Blend Shapes, Forward Kinematics, Linear Blend Skinning, and Vertex Normal calculation—are executed in C++ on the CPU.

The JavaScript wrapper simply feeds the raw inputs (\(\beta, \theta\)) to the WASM memory space and copies the completed, fully-skinned vertices and normals directly into Three.js vertex buffer attributes (VBOs).

Pipeline B: Hybrid WebGL GPU

This pipeline distributes the computational load between the CPU (JavaScript) and the GPU (WebGL via Three.js's SkinnedMesh).

Shape deformations (\(B_S\)), joint regression (\(J\)), and pose-correctives (\(B_P\)) are calculated in JavaScript on the CPU. The resulting mesh coordinates are uploaded to the GPU as the base position attribute. The joint rotations (\(\theta\)) are updated in the skeleton's bone hierarchy. The actual LBS skinning equations are then executed in parallel on the GPU's hardware shaders.

Performance Analysis & Technical Trade-offs

Both approaches have unique bottlenecks that dictate their use cases. The table below outlines the core technical trade-offs between the WebAssembly and WebGL pipelines:

Evaluation Category Pipeline A: WASM CPU Engine Pipeline B: Hybrid WebGL GPU
Math Execution Entirely on CPU (compiled C++). Hybrid: Blendshapes & Regressor on CPU (JS); Skinning on GPU (Shaders).
CPU-to-GPU Upload High: Uploads \(6890 \times 3\) coordinates and \(6890 \times 3\) normals every frame. Low-Medium: Uploads skeleton matrices (\(24 \times 16\) floats) every frame. Base positions only upload when shape/pose parameters change.
Performance (FPS) Bound by CPU compute power and JS-WASM serialization. Typically slower for active sequences. Highly performant (solid 60 FPS) due to parallel GPU vertex transformation.
Multi-Avatar Scalability Poor: Evaluating multiple characters quickly consumes CPU core cycles. Excellent: Multiple instances reuse static VBOs and execute shaders in parallel.
Codebase Synchronization Perfect: Shares the exact C++ source code from h3d.cpp via bindings. Duplicate: Requires maintaining JavaScript ports of SMPL math calculations.
Pose Correctives Impact Low impact (evaluated in highly optimized C++ loops). High impact: JS execution of pose-correctives requires iterating over \(O(N \times 207)\) elements.

Memory and Bandwidth Overhead

The main bottleneck of Pipeline A is the memory bandwidth between the WebAssembly memory heap (HEAPF64) and WebGL. Because Emscripten works within its own isolated memory space, the calculated floating-point coordinates must be copied into a JavaScript typed array before they can be uploaded to the GPU's Vertex Buffer Objects (VBOs).

In code, this data copying is visible in smpl-web/src/main.js: 

// 1. Retrieve raw pointers from WASM Memory space
const ptr = smplRuntime.getVerticesPtr();
const numVertices = smplRuntime.getVerticesSize();

// 2. Instantiate Float64 views directly on Emscripten's Heap
const wasmBufferView = new Float64Array(
  h3d.HEAPF64.buffer,
  ptr,
  numVertices * 3
);

// 3. Copy vertex data from WASM heap to JS Three.js buffer
positions.set(wasmBufferView);
state.geometry.attributes.position.needsUpdate = true;

This process generates substantial serialization overhead when running animations at 30 or 60 frames per second. Conversely, Pipeline B avoids per-frame mesh uploads. Once the base shape is configured (calculated when \(\beta\) changes), the mesh vertices remain static on the GPU. Only the 24 bone transformations (represented as 4×4 matrices or quaternions) are uploaded to WebGL uniforms on each frame: 

// Update bone rotations on CPU in JS
const axis = new THREE.Vector3();
for (let i = 0; i < 24; i++) {
  const rx = pose[i * 3 + 0];
  const ry = pose[i * 3 + 1];
  const rz = pose[i * 3 + 2];
  const theta = Math.sqrt(rx * rx + ry * ry + rz * rz);
  if (theta < 1e-8) {
    bones[i].quaternion.set(0, 0, 0, 1);
  } else {
    axis.set(rx / theta, ry / theta, rz / theta);
    bones[i].quaternion.setFromAxisAngle(axis, theta);
  }
}
// SkinnedMesh automatically updates uniform arrays for GPU skinning

The Pose Correctives Bottleneck

While Pipeline B (GPU) excels at standard skinning, the addition of pose-corrective blend shapes (\(B_P(\theta)\)) introduces a CPU bottleneck in JavaScript. The pose-corrective calculation involves multiplying a 207-dimensional vector (describing the joint orientations) by a large dense matrix of pose directions (\(posedirs\)) for all 6890 vertices.

In JavaScript, this requires a loop over all vertices, which is significantly slower than compiled C++ executed in WASM: 

function updatePoseOffsets(poseFeature, posedirs, out_offsets) {
  const numVertices = 6890;
  // Optimize by only processing dimensions with non-zero coefficients
  const activeDims = [];
  for (let m = 0; m < 207; m++) {
    const coeff = poseFeature[m];
    if (Math.abs(coeff) > 1e-6) {
      activeDims.push({ dim: m, coeff });
    }
  }

  const numActive = activeDims.length;
  if (numActive === 0) {
    out_offsets.fill(0);
    return;
  }

  for (let i = 0; i < numVertices; i++) {
    let ox = 0, oy = 0, oz = 0;
    const pBase = i * 3 * 207;
    for (let d = 0; d < numActive; d++) {
      const active = activeDims[d];
      ox += active.coeff * posedirs[pBase + 0 * 207 + active.dim];
      oy += active.coeff * posedirs[pBase + 1 * 207 + active.dim];
      oz += active.coeff * posedirs[pBase + 2 * 207 + active.dim];
    }
    out_offsets[i * 3 + 0] = ox;
    out_offsets[i * 3 + 1] = oy;
    out_offsets[i * 3 + 2] = oz;
  }
}

When pose correctives are enabled in Pipeline B, the CPU must compute these offsets and upload the updated vertex buffers to the GPU every frame. This reintroduces the VBO upload overhead that GPU skinning was designed to solve, highlighting the classic trade-off between mathematical precision and web performance.

Conclusion

The h3d.cpp project's exploration of web rendering techniques demonstrates that there is no one-size-fits-all solution for real-time character animation. WebAssembly provides a robust, zero-maintenance mechanism to share complex model math directly with high-performance desktop implementations, but faces scaling limits due to browser memory architectures and CPU-GPU bandwidth. Conversely, hybrid GPU skinning offers fluid performance for crowds and avatars, but requires careful optimizations to avoid JavaScript-based CPU bottlenecks when computing complex body deformations like pose-dependent correctives.
